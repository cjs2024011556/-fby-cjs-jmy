# 未完成事项与已知问题

下面把当前项目里还没有完全收口、后续还需要继续补齐的内容整理出来，方便后面接着做。

## 1. 手册数据和向量库还没有完全落地

- 当前项目默认读取 `docs/摩托车发动机维修手册.md`，但这个文件还没有稳定地放到仓库里。
- `docs/` 下的手册正文、图片、HTML 资源还需要继续确认解压位置和文件命名。
- 向量库 `ChromaDB` 目前是按“有手册就自动向量化”的方式做的，但如果手册文件缺失，`vector_chunks` 仍然会是 `0`。
- 后续建议补一个“启动时自动检查手册并自动入库”的独立脚本，避免每次都依赖手工准备数据。

## 2. 数据管理台还可以继续增强

- 现在已经有独立的 `/admin/` 页面，可以做会话和聊天记录的增删改查。
- 但这个页面管理的是本地 JSON 聊天存储，不是一个真正的关系型数据库。
- 当前还没有做分页、排序、全文搜索、导出 CSV/JSON、批量删除等功能。
- 向量库只做了概览展示，还没有做“向量条目级”的可视化管理。
- 后续可以继续增加：
  - 按会话标题搜索
  - 按时间范围筛选
  - 导出单个会话
  - 导出全部数据
  - 查看某条聊天对应的路由理由和章节信息

## 3. 智能体回答链路还有兜底逻辑，说明数据链并不完整

- 现在为了避免空回答，`RAG`、`Workflow`、`Agent` 都加了兜底提示。
- 这说明系统在某些场景下仍然可能没有检索到有效上下文，或者手册数据还没有准备完整。
- 目前的兜底回答是“告诉用户没有加载到可用的维修手册内容”，这能保证页面不空白，但还不算真正的业务回答。
- 后续如果要做得更完整，应该把“空回答兜底”尽量变成“自动补数据 + 真正检索到结果”。

## 4. 还有一些预留接口和预留功能点

- `motoengine/route/admin_router.py` 已经提供了会话和聊天的 CRUD 接口，但后续还可以继续扩展：
  - 用户登录和权限控制
  - 管理员 / 普通用户分级
  - 操作日志审计
  - 数据恢复与回收站
- `motoengine/route/chat_router.py` 已经支持基础聊天和历史查看，但还可以继续补：
  - 聊天内容编辑
  - 单条消息重新生成
  - 会话重命名
  - 更细的流式状态字段
- `motoengine/templates/admin/` 已经是独立页面，但现在更偏“调试管理台”，后面还可以升级成正式的数据后台。

## 5. 代码里还有一些需要后续统一整理的地方

- 项目里有些旧文件仍然能看到中文编码混乱的痕迹，虽然目前运行通过了，但阅读起来不够干净。
- `README.md` 已经重写过，但还可以继续补更简短的“快速开始”说明和截图位置说明。
- `Question.md`、`README.md`、`templates/` 之间的文档说明可以再统一一次，避免以后改了代码却忘了同步文档。
- 现在的测试主要是模块级脚本，还不是完整的自动化测试体系。

## 6. 未来可以继续增加的功能

- 把 `ChromaDB` 做成更完整的可视化浏览页面。
- 增加知识库管理页，把手册章节、手工知识、问答知识分开管理。
- 给 `/admin/` 增加图表统计，比如：
  - 会话趋势图
  - 问题类型分布
  - RAG / Workflow 占比
  - 章节命中统计
- 增加一键初始化脚本，自动完成：
  - 检查 `.env`
  - 检查手册文件
  - 创建向量库
  - 预热数据库
  - 打开首页

## 7. 目前最需要优先确认的点

1. `docs/摩托车发动机维修手册.md` 是否已经真正放到位。
2. `docs/` 和 `motoengine/templates/manual/` 的静态资源是否都能正常打开。
3. `/admin/` 页面是否需要再加权限控制，避免被误删数据。
4. 是否需要把本地 JSON 存储升级成真正的数据库，例如 SQLite 或 PostgreSQL。
5. 是否需要把“空回答兜底”继续优化成“自动重建向量库后再回答”。
